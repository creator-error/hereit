# Wacha Knowledge / Skill Feedback

このメモは、Wacha を使って開発している中で、
なぜ Knowledge や Skill の提案をしなかったのか、
逆にどういう条件なら提案したくなるのかを、
利用者視点の生の所感として整理したものです。

## Overall

今回の開発では、Knowledge / Skill 提案は行いませんでした。

理由は「価値を感じなかった」ではなく、
実装と仕様更新が優先される流れの中で、
提案に必要な知見の安定度と整理度がまだ足りないと判断したからです。

## Why Not

- その時点の主目的が、プロダクト実装を前に進めることだった
- reject 対応や UI 修正が連続し、知見がまだ流動的だった
- Skill / Knowledge にするには、再利用できる形へ要約する必要があった
- 会話で更新された仕様が多く、「これは安定した学び」と言い切れる段階が後ろだった

## Why x2

### 1. なぜ提案より実装を優先したのか

- なぜ 1:
  - 今その場で必要だったのは、タスク完了と差し戻し解消だったからです。
- なぜ 2:
  - Knowledge 提案は価値がありますが、実装の詰まりを直接解消するわけではなく、整理コストが別で発生するからです。

### 2. なぜ知見があってもすぐ提案しなかったのか

- なぜ 1:
  - 途中の気づきと、再利用可能な運用知見は別物だからです。
- なぜ 2:
  - 提案は「思いつき」より「安定した判断軸」として残した方が、後続ユーザーにとって役立つからです。

## When I Would Propose

以下の条件が揃うと、Knowledge / Skill を提案しようと思いやすくなります。

- 同じ説明や同じ修正を 2 回以上繰り返したとき
- reject reason に共通パターンが見えたとき
- 会話での仕様変更をどう吸収すべきか、運用知見として整理できたとき
- 今の Story だけでなく、別 Story でも再利用できると判断できたとき
- ユーザーが「これは知見として残したい」と明示したとき

## Good Point About Wacha Here

- comment と reject reason が残るので、Knowledge 候補を後から掘り起こしやすいです
- Task / Story の流れが残るため、「どこで何を学んだか」を追跡しやすいです

## Weak Point About Wacha Here

- どの時点で Skill / Knowledge 提案へ変換すべきかのトリガーは、現状 worker 側の判断に寄りやすいです
- 実装中に「これは提案候補」と軽く退避する場があると、もっと提案しやすくなるはずです

## Proposals

- `knowledge candidate` のような軽いメモ欄があるとよいです
  - 実装中に見つけた学びを、その場では粗く残せます
- reject や comment から Knowledge 提案候補を拾う補助があるとよいです
  - 提案のし忘れを減らせます
- Story 完了時に「再利用知見はあるか」を確認する導線があるとよいです
  - 提案を自然に思い出せます

## Short Conclusion

Knowledge / Skill を提案しなかったのは、
価値を感じなかったからではなく、
「まだ実装優先で、知見が安定していなかったから」です。

逆に、繰り返しや共通パターンが見え、
再利用可能な形に整理できたときには、
提案したい気持ちはかなり強くなります。
