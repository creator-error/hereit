# Wacha Multi-Agent Feedback

このメモは、Wacha を使って複数エージェントで開発を進める場合に起こりそうな影響を、
利用者視点の生の所感として整理したものです。

前提として、単独 worker でも Wacha は有効ですが、
複数エージェントになると Wacha の価値はむしろ上がる一方、
同時に見えにくい摩擦も増えると考えています。

## Overall

複数エージェント開発では、Wacha の「Task を明示的に分ける力」と
「reject / comment で差分を残す力」がかなり効きます。

ただし、同時並行が増えることで、
Task の粒度、依存関係、最新合意の共有が甘いと一気に破綻しやすくなります。

## Good

- 並列化しやすい
  - 複数 worker が別 Task を同時に進めやすいです。
- 誰が何を触っているか追いやすい
  - `doing` と comment があるので、担当範囲の可視化に向いています。
- reject の価値が上がる
  - 単独開発よりも「何がズレたか」の記録が重要になるため、reject reason が効きます。
- Story を分解する意味が強くなる
  - 単独時より、Task 分割の質がそのまま速度に跳ね返ります。

## Bad

- Task の切り方が悪いと衝突が増える
  - 同じファイルや同じ仕様面を複数 agent が同時に触ると、差分統合コストが急に上がります。
- 最新合意の共有漏れが致命傷になりやすい
  - 1 人なら頭の中で吸収できるズレが、複数人だとそのまま reject や手戻りになります。
- Story 進捗と Task 進捗の差が大きく見える
  - Task は進んでいても、依存する 1 件が詰まると Story 全体は終わりません。
- 「一見進んでいるが実は収束していない」状態が起きやすい
  - 並列実装で画面や API は増えるが、最後に整合性確認でまとめて崩れることがあります。

## Why x2

### 1. なぜ複数エージェントで Wacha の価値は上がるのか

- なぜ 1:
  - 作業の見える化がないと、並列作業の管理コストが急増するからです。
- なぜ 2:
  - 複数人開発の難しさは、実装力そのものより「誰が何をいつどこまで責任を持つか」の管理にあるからです。Wacha はそこを補助できます。

### 2. なぜ同時に摩擦も増えるのか

- なぜ 1:
  - Task の境界と仕様の境界がずれていると、複数 agent が同じ問題を別方向から解いてしまうからです。
- なぜ 2:
  - 並列化は速度を上げますが、前提共有が曖昧なまま進むと「速く間違う」だけになるからです。

## Expected Impact

### 良い影響

- UI、API、repository、docs のように責務を分ければ、かなりスピードが出るはずです。
- reviewer が reject するときも、どの Task にどの理由を返すか整理しやすくなります。
- manager が Story を分解する価値が明確になります。

### 悪い影響

- 依存関係が強い Task を並列に投げると、待ちや手戻りが増えます。
- 口頭での仕様変更が多いと、更新を見ていない worker が古い前提で進めやすいです。
- 「どこまで完成なら review に出してよいか」が曖昧だと、in_review の質が下がります。

## Proposals

- Task に `owner scope` を明示したいです。
  - 例: `UI only`, `API only`, `repository only`, `docs only`
- Task に `depends on` を軽く持たせたいです。
  - 並列着手してよいかが分かりやすくなります。
- Task 文面とは別に `latest-agreement` を持たせたいです。
  - 口頭更新の反映先があるだけで、並列時のズレはかなり減ります。
- review 前提を段階化したいです。
  - `skeleton`, `functional`, `production-ready` のような段階があると、並列作業の出口が揃います。
- reject を Task 単位だけでなく Story 観点でも見たいです。
  - Story 全体のボトルネック把握がしやすくなります。

## Short Conclusion

複数エージェント開発では、Wacha の有効性は高まります。
特に Task 分割、担当の見える化、reject reason の記録は強みです。

一方で、前提共有の弱さも拡大します。
そのため、単独開発以上に
「Task 境界の明確さ」
「最新合意の共有」
「完了レベルの定義」
が重要になると考えます。
