# Wacha Worker Feedback

このメモは、Codex worker として Wacha を使いながら開発を進めたときの所感を、
改善提案込みで端的に整理したものです。

## Overall

Wacha は「今どの粒度の仕事を、どの状態で進めているか」を明確に保ちやすく、
複数の実装・レビュー・差し戻しが並行する状況で特に強いです。

一方で、実装が口頭で更新されながら進むケースでは、
Task 文面と実際の合意内容がずれていくことがあり、
worker 側でその差分を吸収する運用負荷が出やすいです。

## Good

- Task を `todo` / `doing` / `in_review` / `wait_accept` / `accepted` / `rejected` で分けられるのは明快です。
- reject reason が残るので、差し戻し対応の再現性が高いです。
- Story 単位でまとまりを持たせつつ、実装は Task 単位で細かく進められるのが実務に合っています。
- worker がコメントを積み上げられるため、「何を変えたか」をレビュー前に共有しやすいです。

## Bad

- Task の完了条件が会話中の方針変更に追従しないと、reject が増えやすいです。
- worker は Story を直接前進させにくく、Task は進んでいても Story 全体の見通しが少し切れます。
- UI 先行で見せたい場面では、「骨格を見せる段階」と「API まで含めて完了とする段階」が Task 文面から読み取りにくいことがあります。
- role 制約の都合で、必要な Story/Task にすぐ触れない場面があり、流れが一瞬止まります。

## Why x2

### 1. なぜ reject が増えやすいのか

- なぜ 1:
  - 口頭での仕様更新が速く、Task 文面より会話の最新合意が先に進むからです。
- なぜ 2:
  - Wacha の Task は「着手時点の要件固定」には強い一方、「会話で流動的に変わる要求」の更新コストを worker が肩代わりしやすいからです。

### 2. なぜ Wacha はそれでも有効なのか

- なぜ 1:
  - reject reason と comment が残るため、途中で方針が揺れても、どこで何がずれたかを後から追えるからです。
- なぜ 2:
  - 開発の本質は「一度で正解すること」より、「差分を明示しながら合意に収束させること」だからです。Wacha はその収束過程の記録に向いています。

## Proposals

- Task 文面とは別に `latest-agreement` のような短い更新欄があると良いです。
  - 会話での最新合意だけを上書きで残せると、reject の主因を減らせます。
- Task に `UI skeleton` と `production-ready` のような完了レベルを持たせたいです。
  - UI を先に見せたい開発で、どこまでやれば review に出してよいかが明確になります。
- Story の進捗サマリを worker から見やすくしたいです。
  - どの Task を終えると Story のボトルネックが解消するかが、今より判断しやすくなります。
- role 制約エラーは「何の role が必要か」をもう少し明示するとよいです。
  - 進行が止まったときに、worker が次の打ち手を決めやすくなります。

## Short Conclusion

Wacha は、実装とレビューを往復しながら進める開発フローにかなり相性が良いです。
特に reject reason と comment の組み合わせは強いです。

改善余地は、「会話で更新された最新合意を、Task 文面と別で軽く保持すること」と、
「完了レベルの粒度を UI 骨格 / 実運用実装で分けること」にあります。
